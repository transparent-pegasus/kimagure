# リリースノート作成スキル

このスキルは、前回のリリース以降に加えられた変更を分析し、ユーザーに向けた分かりやすいリリースノートを作成するための手順です。

## 手順

1.  **前回リリースの特定**
    *   `docs/releases/` ディレクトリを確認し、最新のリリースノートファイルを特定します。
    *   そのファイルの日付またはバージョンを確認します。

2.  **変更点の抽出**
    *   前回のリリース以降の変更内容を調査します。
    *   `docs/todo.md` の完了済みタスク (`[x]`) や、最近のコミットログ (`git log`) を参照します。
    *   **重要**: ユーザー体験（UX）に影響を与える変更のみを抽出してください。
        *   ✅ 対象: 新機能、デザイン変更、使い勝手の改善、バグ修正（ユーザーに見えるもの）。
        *   ❌ 除外: リファクタリング、ビルド設定の変更、内部ライブラリの更新、コード整形、テストコードの修正など。

3.  **リリースノートの作成**
    *   新しいファイル `docs/releases/YYYYMMDD_vX.Y.md` を作成します（バージョン番号は適宜インクリメント）。
    *   以下の構成で記述します：
        *   **タイトル**: `きまぐれごはん vX.Y リリースノート`
        *   **概要**: 今回の更新のハイライトを数行で。
        *   **主な変更点**: カテゴリ分けして箇条書き（例: 「新機能」「改善」「修正」）。
        *   **日付**: リリース日。

4.  **内容の推敲**
    *   専門用語を避け、ユーザーにメリットが伝わる言葉を選びます。

## ルール

*   **技術的詳細の排除**: 「APIのエンドポイント修正」「Reactのフック修正」などは書かず、「動作が安定しました」等の表現に留めるか、UXに影響なければ記載しない。
*   **ポジティブな表現**: 「バグを直した」だけでなく、「〇〇が正しく動くようになりました」のように記述する。
